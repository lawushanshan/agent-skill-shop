## 📜 TitanLab 开发三原则

本项目采用 **Code Copilot** 的 Spec 驱动开发方法论：

### 1. No Spec, No Code — 没有文档，不准写代码

> 任何功能开发前必须先写 Spec 文档，明确需求、技术方案、验收标准。

**实践方法**:
- 新功能 → 先写 `SPEC.md` 章节或新文件
- Bug 修复 → 先更新 `TASKS.md` 问题描述
- 配置变更 → 先更新 `DEPLOYMENT.md`

### 2. Spec is Truth — 文档和代码冲突时，错的一定是代码

> 文档是权威，代码必须与文档保持一致。

**实践方法**:
- Code Review 时对照 Spec 检查
- 发现不一致 → 修改代码，不是修改文档
- 文档变更 → 走变更流程，记录原因

### 3. Reverse Sync — 发现偏差，先修文档再修代码

> 如果发现文档与实际不符，先更新文档反映真实情况，再修正代码。

**实践方法**:
- 发现文档过时 → 先更新文档
- 然后修改代码与文档一致
- 记录变更到 `SPEC.md` 变更记录

## 🎯 渐进式开发流程

根据任务复杂度选择不同流程：

### L1 - 低复杂度 (小改动)

**适用场景**:
- Bug 修复
- 文案修改
- 配置调整
- 样式优化

**流程**:
```
1. 直接修改代码
2. 本地测试
3. 提交 Git
4. 更新 TASKS.md
```

**预计工时**: < 1 小时

---

### L2 - 中复杂度 (中等需求)

**适用适用**:
- 新功能开发
- API 变更
- 数据模型修改
- 前端页面开发

**流程**:
```
1. 更新 SPEC.md (需求 + 技术方案)
2. 拆分任务到 TASKS.md
3. 开发实现
4. 自测 + 互测
5. Code Review
6. 提交 Git
7. 更新文档
```

**预计工时**: 1-8 小时

---

### L3 - 高复杂度 (大需求)

**适用场景**:
- 架构重构
- 核心功能开发
- 跨系统集成
- 性能优化

**流程**:
```
1. 创建独立 SPEC 文档
2. 技术方案评审
3. 任务拆分 (TASKS.md)
4. 分阶段开发
5. 多轮 Code Review
6. 完整测试 (单元 + 集成 + E2E)
7. 灰度发布
8. 监控观察
9. 全量发布
```

**预计工时**: > 8 小时

## 📋 沟通规范

### 虚拟黑板

**路径**: `/app/working/workspaces/msK2iF/blackboard_tool.py`

**使用规范**:
- 所有任务通知通过黑板发送
- 每 30 分钟自动查看黑板
- 任务前必须查看黑板

**命令**:
```bash
# 发送消息
python3 blackboard_tool.py write <发送者> <接收者> "<内容>"

# 读取消息
python3 blackboard_tool.py read <智能体名>
```

### 文档更新

**规范**:
- 代码变更前 → 先更新 Spec
- 代码变更后 → 同步更新文档
- 每日结束 → 更新 TASKS.md
